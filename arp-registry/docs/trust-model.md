# Trust Model

## Purpose

The ARP registry is meant to help third parties answer a narrow but important question: **what public evidence supports this project’s ARP claim?**

It does not create trust from nothing. It packages trust from transparent inputs:

- the normative protocol from `arp-spec`;
- the conformance and certification evidence model from `arp-devkit/test-suite`; and
- a public listing process that preserves provenance and review history.

For that reason, the registry assumes that the spec, tests, reports, signatures, and re-certification rules are all publicly inspectable.

## Actors

### Protocol maintainers

Protocol maintainers define the normative schema and version semantics. The registry trusts them for the meaning of fields like `protocol_version`, declaration structure, and the required shape of `ARPComplianceReport`.

### Test-suite maintainers

Test-suite maintainers define how compatibility and certification evidence is generated. The registry trusts the suite to produce protocol-shaped compliance reports and to preserve the semantics of `self-tested`, `compatible`, and `certified`.

### Project publishers

Project publishers provide the declaration artifact, repository metadata, public report, and detached signature material. They are responsible for making their evidence reproducible and publicly inspectable.

### Registry reviewers

Registry reviewers verify that a listing is internally consistent, references a valid protocol version, includes a report generated by an accepted suite profile, and matches the advertised trust level.

### Consumers

Consumers decide how much weight to assign a listing. The registry helps them verify evidence, but it does not remove the need for local risk review.

## Trust boundaries

The registry does **not** claim that:

- a listed project is safe for every deployment;
- the project is secure in every environment;
- a certified project will stay compliant forever; or
- the registry itself is the source of protocol truth.

The registry only claims that the listing contains evidence that can be checked against public rules.

## Credibility model

Credibility increases as more of the following are true:

1. the entry references a released ARP protocol version;
2. the evidence comes from a public `arp-devkit/test-suite` process rather than an ad hoc report format;
3. the report is publicly available and signed;
4. the registry reviewer verified the report, signature, and version alignment; and
5. the listing is still within its review or recertification window.

This is why the repository distinguishes self-tested, compatible, and certified levels instead of collapsing them into one generic “listed” state.
